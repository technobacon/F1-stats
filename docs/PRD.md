# PRODUCT REQUIREMENTS DOCUMENT (PRD)
## Project: F1 StatGuesser (Working Title)
**Document Version:** 1.0
**Target Audience:** Frontend Developers, Backend/Data Engineers, UI/UX Designers, QA Engineers

## 1. Executive Summary
F1 StatGuesser is a web-based, gamified sports trivia platform built for Formula 1 fans. The application leverages high-accuracy historical data to present users with numerical guessing challenges, scoring them based on their proximity to the true statistic. The platform utilizes a "hybrid-scarcity" retention model, blending capped daily challenges with an unlimited, highly bingeable arcade mode to maximize user engagement and ad-impression inventory.

## 2. Mathematical Scoring Engine
For all exact-numerical guessing modes, developers must implement a percentage-error exponential decay algorithm. This normalizes the score regardless of whether the target value is small (e.g., 7 World Championships) or large (e.g., 1,075 Career Points).

The absolute maximum score per question is **5,000 points**. The minimum score is **0 points**.

The formula to calculate the score for any given question is defined as:

### Parameter Definitions
- **Guess**: The raw numerical value inputted by the player.
- **Actual**: The true statistical value fetched from the validated database.
- **λ (Lambda)**: The structural severity multiplier (float). This variable dictates how punishing an incorrect guess is.
  - *Default Setting:* Set λ = 3.0.
  - *Example Impact:* With λ = 3.0, a player who misses the target by exactly 10% will receive approximately **3,704 points**. A player who misses by 50% will receive approximately **1,116 points**.
- **Edge Case Handling:** If Guess = Actual, the score must resolve instantly to **5,000**. If Actual = 0 and Guess > 0, the score drops to **0** to prevent division-by-zero errors.

## 3. Core System Architecture & Data Pipeline

```
[ Jolpica / OpenF1 APIs ] ──► [ Asynchronous Scripts ] ──► [ LLM Question Generator ]
                                                                     │
[ Production Database ]   ◄── [ Automatic Validation Script ] ◄──────┘
```

### 3.1 Data Sourcing
- **Primary Source:** Historical telemetry, driver, constructor, and circuit statistics must be integrated using the **Jolpica F1 API** (api.jolpi.ca).
- **Secondary Source:** Modern live timing and tire telemetry data can optionally expand into the **OpenF1 API** (openf1.org).

### 3.2 Automated Question Generation Pipeline (Anti-Hallucination)
To protect user experience, data must **never** be fetched live from the external APIs during consumer gameplay. Developers must build an offline ETL (Extract, Transform, Load) pipeline:

1. **Extraction:** An isolated backend script pulls raw tables (wins, poles, fast laps, circuit records) from the Jolpica API weekly.
2. **LLM Synthesizer:** The raw JSON is fed into an internal LLM prompt pipeline to generate engaging phrasing for 1,000+ questions.
3. **Strict Validation Layer (Critical):** Before an AI-generated question is committed to the production database, an automated script must independently execute a deterministic SQL/code query to verify that the target integer generated by the LLM precisely matches the true math in the raw Jolpica dataset. If a discrepancy exists, the question is discarded.

## 4. Feature Breakdown & Gameplay Modes

### 4.1 Technical Matrix of Game Modes
The frontend application must support four distinct operational views:

| Mode Name | Input Type | Frequency Limit | Technical Dependency |
|---|---|---|---|
| **Daily General Quiz** | Precise Text Box + Slider | 1 Session (5 Questions) per 24 hours | Global cron job reset at 00:00 UTC |
| **Race-Week Quiz** | Precise Text Box + Slider | 1 Session (5 Questions) per active race week | Conditioned on real-world F1 weekend schedules |
| **The One-Shots** | Hardcore Text Input | Max 3–5 unique questions per 24 hours | Tracks individual user attempt arrays |
| **Arcade Over/Under** | Twin Button Clicks (A/B) | **Unlimited** | Algorithmic pairing matrix of database entities |

### 4.2 The Arcade Over/Under Matrix Engine
To ensure infinite replayability without manual question writing, the backend must dynamically generate "Who Has More?" pairs.

- **The Logic:** Select two random drivers from a shared era and pull a matching metric.
- *Example:* Entity_A (Alonso, Podiums: 106) vs Entity_B (Räikkönen, Podiums: 103).
- The UI displays the names and a button for each. Clicking a button triggers an instant boolean evaluation (Score++ if correct, reset streak if incorrect).

## 5. UI/UX & Gamification Specifications

### 5.1 Dynamic Countdown Header (HUD)
- **Location:** Fixed to the global top navigation bar.
- **Behavior (In-Season):** Displays an active countdown timer (DD : HH : MM : SS) ticking down to the closest upcoming session based on the official FIA calendar API (FP1, FP2, FP3, Qualifying, or Race).
- **Behavior (Off-Season):** Automatically pivots its target destination text to display the countdown to the first team car launch, pre-season testing day, or Round 1 Lights Out.

### 5.2 Storage Architecture ("Guest-First" Authentication)
To completely minimize friction and optimize top-of-funnel conversion:

- All player scores, historical accuracy tracking, streaks, and unlocked achievement flags must write directly to the browser's `window.localStorage` by default.
- **Account Wall Trigger:** The app must only prompt user registration (Email, Google Auth) when a user attempts to post their score to the Global Leaderboard, or clicks "Sync Progress Across Devices."

### 5.3 Tribal Theming & Profile Customization
- Upon account creation, the user selects their favorite current or legacy Constructor.
- Selecting a team alters global CSS variables to apply custom accent colors to UI borders, buttons, data graphics, and the player profile card (e.g., `#EF3829` for Ferrari, `#6CD3BF` for Mercedes, `#FF8700` for McLaren).
- **Constructors Championship Leaderboard:** The backend must aggregate all points scored by users assigned to a specific team faction, creating a secondary, live global standings chart showcasing which fanbase is statistically the smartest.

## 6. Monetization & Ad Placement Architecture
To preserve a clean, fast gameplay loop while maximizing programmatic yield (RPM), ads must be completely decoupled from active thinking phases.

- **Allowed Ad Zones:**
  1. **The Reveal Transition:** A small, standardized banner anchored at the baseline of the view screen while the scoring engine runs its calculation animation.
  2. **The Finish Summary Screen:** A high-visibility display ad integrated into the final scorecard view where players review stats and copy their shareable results markup.

- **Ad Network Implementation:** The frontend wrapper must natively support header bidding integration. Standard Google AdSense must be swapped out for HTML5 game-centric monetization layers (such as *AdInPlay* or *Playwire*) to ensure programmatic video fill and optimized CPM rates from Tier 1 traffic zones (US, UK, Western Europe).

- **Premium Feature Gate:** A backend flag `is_premium: true` must bypass all ad containers and grant unrestricted database access to the historic quiz archive and deep-dive user performance analytics dashboards.
